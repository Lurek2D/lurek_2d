    1. What Lurek2D is¶

    One binary, one scripting language, one afternoon to learn. Source under src/ is organised into five responsibility groups â€” Foundations, Core Runtime, Platform Services, Feature Systems, and Edge/Integration â€” defined in architecture/philosophy.md and inventoried in architecture/engine-architecture.md. Renderer is wgpu (Vulkan / DX12 / Metal); physics is rapier2d; audio is rodio. Engine version is 1.0.0 (Cargo.toml). Licence is MIT. Desktop targets only â€” Windows, Linux, macOS on x86_64 and ARM (constraint A-02).

    3. First 30 minutes¶

    A working dev loop in five steps.

    1. Install Rust â‰Ą 1.78 (stable). The repo pins stable via rust-toolchain.toml; rustup will pick it up automatically. You also need Python â‰Ą 3.10 for the docs / validation tooling.
    2. Clone: 
       git clone https://github.com/LurekDude/lurek_2d.git
 cd lurek_2d
    3. Build & run the splash screen: 
      cargo run
      The first build pulls and compiles the dependency tree (â‰3â€“8 minutes on a modern laptop). After that, incremental builds finish in seconds. The splash window opens with no game loaded.
    4. Run a showcase game: 
      cargo run -- content/games/showcase/hello_world

    5. Open VS Code and accept the workspace extension recommendations (extension/vscode/).     5. Build and run¶

    Cargo profiles:

    • cargo build â†’ build/debug/lurek2d â€” fast incremental, slow runtime, full debug symbols.
    • cargo build --release â†’ build/release/lurek2d â€” slow build, optimised binary (target is â‰¤ 10 MB once the plugin architecture lands).
    • cargo check â€” type-check only; the fastest "did I break the build?" loop.

    The build/ directory is the configured Cargo target-dir (.cargo/config.toml). Don't use target/.

    Rust/build config has a small set of fixed-location files that must stay where the toolchain expects them: Cargo.toml, build.rs, rust-toolchain.toml, Cargo.lock, and .cargo/config.toml. Support config discovered by Cargo subcommands also lives under .cargo/; for example, cargo-nextest reads .cargo/nextest.toml. Do not move these into a custom config/ folder.

    Feature flags:

    • Default: lua-jit â€” vendored LuaJIT 2.1 via mlua (constraint B-01).
    • Fallback: cargo build --no-default-features --features lua54 â€” Lua 5.4 for environments where LuaJIT cannot vendor.

    6. Writing your first game¶

    A Lurek2D game is a folder with a main.lua (mandatory) and an optional conf.lua. The engine can call several optional lifecycle callbacks; the most common starting trio is lurek.init, lurek.process, and lurek.draw (constraint C-04: an empty main.lua is a valid game).

    Smallest possible game â€” content/games/showcase/my_game/main.lua:

    function lurek.init()
    -- One-time setup. Load assets, init state.
    state = { x = 100, y = 100 }
end

function lurek.process(dt)
    -- Per-frame logic. dt is seconds since last frame.
    if lurek.input.keyboard.isDown("right") then
        state.x = state.x + 200 * dt
    end
end

function lurek.draw()
    -- Per-frame render. Issue draw commands here.
    lurek.render.setColor(1, 1, 1, 1)
    lurek.render.print("Move with â†’ key", 20, 20)
    lurek.render.rectangle("fill", state.x, state.y, 32, 32)
end

    Add a conf.lua next to it for window setup (optional):

    function lurek.conf(t)
    t.window.title  = "My Game"
    t.window.width  = 800
    t.window.height = 600
end

    Run it:

    cargo run -- content/games/showcase/my_game

    Or build once and drag the my_game/ folder onto the splash window. Browse the full lurek.* surface in docs/api/lurek.md. For Lua patterns and idioms, see the lua-scripting skill.

    Adding assets. Put a PNG next to main.lua and load it in lurek.init:

    function lurek.init()
    state = { x = 100, y = 100 }
    state.player = lurek.render.newImage("player.png")
    state.click   = lurek.audio.newSource("click.ogg", "static")
end

function lurek.draw()
    lurek.render.draw(state.player, state.x, state.y)
end

function lurek.mousepressed(x, y, button)
    state.click:play()
end

    Assets are resolved relative to the game folder by GameFS. The sandbox prevents .. escapes. Supported image formats include PNG (always), JPEG, GIF, and DDS (compressed). Audio formats are OGG Vorbis and WAV â€” see Cargo.toml for the build-time format set. For more single-API examples, browse content/examples/README.md.

    7. Writing your first engine change¶

    Workflow for a one-bug-or-feature PR against src/.

    1. Pick a module under src/ (e.g. src/timer/).
    2. Read its spec at docs/specs/timer.md â€” types, functions, Lua API surface, current test paths. The spec is the canonical contract.
    3. Make the Rust change. Follow the file-structure rules in architecture/engine-architecture.md Â§ Module Internal File Structure Standard: thin mod.rs, <primary>.rs for logic, no impl LuaUserData outside src/lua_api/, no wgpu::* outside src/render/. Every new pub item gets a /// doc comment (constraint Q-05).
    4. Regenerate auto-spec sections: 
      python tools/gen_all_docs.py
      This rewrites the Files / Types / Functions / Lua API Reference / References sections of every affected spec and refreshes docs/api/lurek.md, docs/api/rust.md, docs/api/lureksome.md, docs/api/lurek.lua, and docs/api/lureksome.lua.
    5. Update the spec's manual sections (## Summary, ## Notes) only if behaviour changed in a user-visible way.
    6. Add tests â€” Lua first. The Lua-first rule is binding: behaviour observable through lurek.* MUST be tested in Lua under tests/lua/, and the new test file must be registered manually in tests/lua/harness.rs. Reach for tests/rust/unit/ only for internals not reachable from Lua.
    7. Run quality gates (Â§10).
    8. Commit with type(scope): description â€” see Â§10 for format.

    For a deeper walkthrough including the Thin Wrapper Rule for new lurek.* APIs, load the lua-rust-bridge and rust-coding skills.

    9. Testing¶

    The Lua-first testing rule is binding (see philosophy.md Â§ Testing Constraints, constraints TST-01..TST-04): behaviour observable through lurek.* is tested in Lua. Rust unit tests under tests/rust/unit/ are reserved for internals not reachable through Lua bindings.

    Testing rules for contributors¶

    • Write Lua tests by default. If your change affects anything a game script can see, test it under tests/lua/.
    • Rust unit tests are for internals only. Put them in tests/rust/unit/_tests.rs. Never add #[cfg(test)] blocks inside src/.
    • Keep src/lua_api/*_api.rs and every mod.rs thin. Business logic goes in src/<module>/*.rs sibling files. See TST-03 and TST-04 in philosophy.md Â§ Testing Constraints.

    Full decision tree and banned-patterns list: test-framework.md Â§ Test placement.

    Two test layers (full design in architecture/test-framework.md):

    • Rust â€” cargo test. Per-module suites under tests/rust/unit/<module>_tests.rs. Float comparisons use absolute tolerance, never assert_eq! on f32.
    • Lua BDD â€” cargo test lua_test_<category>_<name>. Files under tests/lua/ using describe / it / expect_equal from tests/lua/init.lua. Every Lua test file must end with test_summary().

    Adding a new Lua test:

    1. Create tests/lua/<category>/test_<name>.lua.
    2. Add a matching #[test] fn lua_test_<category>_<name>() entry in tests/lua/harness.rs â€” auto-discovery is intentionally not used.
    3. Run: cargo test lua_test_<category>_<name> -- --nocapture.

    Common test workspace tasks: Test: All, Test: Math module, Test: Physics module, Test: Graphics module, Test: Audio module, Test: Input module, Test: Lua bindings. See Â§5.

    Evidence vs golden suites: evidence files produce artefacts (screenshots, logs); golden files are compare-only baselines. Audit with python tools/audit/lua_evidence_golden_contract_audit.py.

    Skeleton of a Lua BDD test:

    local t = require("tests.lua.init")

t.describe("timer", function()
    t.it("counts up monotonically", function()
        local first  = lurek.timer.getTime()
        local second = lurek.timer.getTime()
        t.expect_true(second >= first)
    end)
end)

t.test_summary()

    For Rust tests, place per-module suites under tests/rust/unit/<module>_tests.rs. Name tests <subject>_<scenario>_<expected> (no test_ prefix). Float comparisons must use absolute tolerance: assert!((actual - expected).abs() < 1e-5).

    10. Quality gates and commits¶

    From the system prompt's Quality Gates section (.github/copilot-instructions.md) â€” run all of these before any commit:

    • Commit format: type(scope): description where type â {feat, fix, refactor, test, docs, chore}. One logical change per commit.

    Before pushing checklist:

    • [ ] All quality gates green locally.
    • [ ] CHANGELOG entry added under the current version.
    • [ ] Specs regenerated if you touched src/<module>/*.rs or src/lua_api/*_api.rs.
    • [ ] Lua test added if you added or changed a lurek.* function (constraint C-04).
    • [ ] Rust integration test added if you added a pub fn / pub struct / pub enum / pub trait (constraint Q-03).

    11. Working with AI agents (CAG)¶

    The CAG layer (Context Augmented Guidance) lives under .github/ and customises how Copilot â€” or any agent honouring Markdown system prompts â€” works in this repo. Five artifact types: the system prompt, agents, skills, prompts, and companion files.

    • System prompt (.github/copilot-instructions.md) is the only file always loaded; it teaches the agent how to find everything else.
    • Agents under .github/agents/ are workflow specialists (Architect, Developer, Doc-Writer, Tester, â€¦). Each declares which personas it serves.
    • Skills under .github/skills/<name>/ are deep domain knowledge loaded on demand when the user task matches the skill's description.
    • Prompts under .github/prompts/<verb>-<noun>.prompt.md are user-invocable playbooks, surfaced as slash commands. Example: /run-quality-sweep, /audit-module, /create-demo.

    Full deep dive in architecture/cag-system.md. When working with an agent on a multi-step change, route through Manager â€” it engages Planner for â‰Ą 3-agent or â‰Ą 5-file tasks before any implementation.

    Common slash prompts â€” each one is a parameterised playbook under .github/prompts/:

    Prompt What it does
    /run-quality-sweep Runs the full audit â†’ diagnose â†’ fix â†’ verify cycle across the repo.
    /run-quality-gates Equivalent to the Quality Gate: Full task: fmt + clippy + test.
